# Описание проекта: диплом1c

## Общая структура

- **client/** — фронтенд (React, Vite, Bun, Nginx)
- **server/** — бэкенд (NestJS, Prisma, PostgreSQL)
- **docker-compose.yml** — запуск всего стека
- **uploads/** — папка для загружаемых файлов

---

## Фронтенд (client)

### Технологии
- **React** (функциональные компоненты, хуки)
- **Vite** (сборка и dev-сервер)
- **Bun** (используется для сборки в Dockerfile)
- **Nginx** (проксирование и отдача статики)
- **ESLint** (линтинг)
- **react-router-dom** (маршрутизация)
- **axios** (работа с API)
- **react-hot-toast** (уведомления)
- **react-icons**, **leaflet**, **react-leaflet** (UI и карты)

### Архитектура
- **App.jsx** — маршрутизация, контекст авторизации, Toaster
- **contexts/AuthContext.jsx** — хранение пользователя, методы login/register/logout
- **api/** — отдельные файлы для auth, cart, messages, user, axios instance с интерцепторами
- **hooks/** — хуки для загрузки данных (useProducts, useServices)
- **utils/cookies.js** — работа с куками (токен, пользователь)
- **components/** — UI-компоненты, модальные окна, защищённые маршруты
- **pages/** — страницы приложения (Home, Catalog, Services, About, Contacts, Profile, Checkout, Admin, Chat)
- **Dockerfile** — двухэтапная сборка (Bun → Nginx)
- **nginx.conf** — проксирование /api на backend

### Маршрутизация
- Открытые: `/`, `/catalog`, `/services`, `/about`, `/contacts`
- Защищённые (ProtectedRoute): `/profile`, `/checkout`, `/admin`, `/chat`

### Пример страницы
- **HomePage** — секции: MainSection, PopularProduct, InfoSection, VideoBlock, ContactSection, Footer
- **AdminPage** — управление продуктами/услугами, проверка роли пользователя (admin), работа с API

## Бэкенд (server)

### Технологии
- **NestJS** (структурированный сервер на TypeScript)
- **Prisma ORM** (работа с PostgreSQL)
- **Docker** (контейнеризация)
- **Модули:** auth, cart, messages, products, requests, services, upload, users
- **Prisma schema** — описание моделей, миграции, seed

### Архитектура
- **src/** — модули, контроллеры, сервисы, DTO, guards, стратегии
- **prisma/** — schema.prisma, миграции, seed.ts
- **Dockerfile** — сборка backend
- **entrypoint.sh** — скрипт инициализации

## Порядок запуска и деплоя

1. **Клонировать репозиторий**
2. **Настроить .env файлы** для client и server (API URL, DB URL и т.д.)
3. **Запустить docker-compose**:
   ```sh
   docker-compose up --build
   ```
4. **Backend**:
   - Автоматически применяются миграции Prisma
   - Запускается NestJS сервер на порту 3001
5. **Frontend**:
   - Собирается через Bun/Vite
   - Отдаётся через Nginx (порт 80)
   - /api проксируется на backend
6. **Проверить доступность:**
   - http://localhost (frontend)
   - http://localhost:3001 (backend API)

## Подробное проектирование доработки: аналитика и отчётность

### Цели доработки
- Дать администратору инструменты для анализа ассортимента: какие товары наиболее популярны, сколько раз их покупали и просматривали.
- Повысить прозрачность и управляемость ассортимента.

### Технологии для реализации
- **Backend:**
  - NestJS (REST API, сервисы, контроллеры, guards)
  - Prisma ORM (моделирование данных, миграции, запросы)
  - PostgreSQL (хранение статистики)
- **Frontend:**
  - React (страницы, компоненты, хуки)
  - Axios (запросы к API)
  - Chart.js или Recharts (визуализация графиков)
  - React Table (таблицы)

---

### Проектирование backend-части

#### 1. Моделирование данных (Prisma)
- В модель Product добавить поля:
  - `purchaseCount` (int, default 0) — количество покупок
  - `viewCount` (int, default 0) — количество просмотров
- (Рекомендуется) Добавить модель Order или Purchase для хранения истории покупок:

```prisma
model Product {
  id           Int      @id @default(autoincrement())
  name         String
  ...
  purchaseCount Int     @default(0)
  viewCount     Int     @default(0)
  purchases     Purchase[]
}

model Purchase {
  id         Int      @id @default(autoincrement())
  product    Product  @relation(fields: [productId], references: [id])
  productId  Int
  user       User?    @relation(fields: [userId], references: [id])
  userId     Int?
  createdAt  DateTime @default(now())
}
```

#### 2. API и сервисы
- **POST /products/:id/view** — инкрементировать просмотры товара
- **GET /products/popular?limit=10** — получить топ популярных товаров по покупкам
- **GET /products/:id/stats** — получить статистику по просмотрам и покупкам
- **GET /products/sales?from=...&to=...** — динамика продаж за период
- (Опционально) **GET /products/views?from=...&to=...** — динамика просмотров

#### 3. Логика
- При оформлении заказа увеличивать `purchaseCount` у соответствующих товаров и/или создавать записи в Purchase.
- При открытии карточки товара (или страницы) вызывать эндпоинт для увеличения `viewCount`.
- Для отчётности использовать агрегирующие запросы Prisma (count, groupBy).

### Пример сценария работы
1. Пользователь открывает карточку товара — фронт отправляет POST /products/:id/view.
2. При оформлении заказа — backend увеличивает purchaseCount и/или создаёт запись в Purchase.
3. Админ открывает раздел "Аналитика" — фронт запрашивает топ товаров и статистику, отображает таблицы и графики.

---

### Безопасность и производительность
- Эндпоинты аналитики доступны только для пользователей с ролью admin (guard на backend).
- Для избежания накрутки просмотров — можно ограничить инкремент по IP/пользователю/времени (опционально).
- Для больших данных — использовать пагинацию и агрегацию на сервере.

---

> Такой подход позволит гибко масштабировать аналитику, расширять отчётность и интегрировать новые метрики по мере роста проекта.
